# COMBAT_CORE_V1_SPEC.md

## Статус документа

Это рабочая спецификация новой основы боевой системы для RPG-бота.

Цель документа:
- зафиксировать **единую модель боя**, на которую можно опираться при будущих изменениях;
- дать понятный ориентир для переписки `combat.py`, `skill_engine.py`, `balance.py`;
- снизить риск случайных регрессий, когда меняются оружия, эффекты, формулы или структура хода;
- быть удобным референсом и для человека, и для ИИ.

Документ **не** задаёт финальный баланс всех оружий и всех чисел.
Он задаёт **архитектуру боевого ядра** и общие правила, по которым потом будут строиться формулы и механики.

---

## 1. Почему нужен новый combat core

Текущее состояние проекта уже достаточно стабилизировано рефакторами и regression tests, но в ядре боя остаются системные проблемы:

1. Обычная атака и skill-flow до сих пор проходят по разным логическим путям.
2. Баффы и дебаффы хранятся частично как список эффектов, а частично как отдельные флаги в `battle_state`.
3. Некоторые эффекты имеют описание с длительностью, но в runtime не имеют нормального счётчика ходов.
4. Архитектура боя пока хуже масштабируется под:
   - будущие оружия;
   - будущие эффекты;
   - групповой бой;
   - переосмысление баланса.

Из-за этого менять боевую систему дальше уже неудобно и рискованно.

---

## 2. Что является целью этого этапа

### Главная цель
Сделать **Combat Core v1** — новую основу боевой системы, где:
- normal attack и damage-skill подчиняются одной модели расчёта;
- timed-эффекты имеют нормальную длительность и предсказуемый lifecycle;
- боевой ход разбит на понятные фазы;
- `handlers/battle.py` отвечает за UI/оркестрацию, а не за боевую математику;
- на эту основу потом можно безопасно накладывать новую философию оружий из `GAME_FOUNDATION.md`.

### Что мы хотим получить в результате
После завершения этого этапа должно быть проще:
- менять формулы в `balance.py`;
- менять смысл оружий и скиллов;
- добавлять новые эффекты;
- вводить групповой бой;
- ловить регрессии и понимать, где именно что сломалось.

---

## 3. Что НЕ входит в этот этап

Чтобы не расползтись по scope, в Combat Core v1 **не входят**:

- полная переделка всех оружий;
- финальная балансировка цифр;
- полный групповой бой;
- новые типы экипировки;
- новые деревья навыков;
- полный UI-редизайн боя.

То есть это этап **архитектуры боевого ядра**, а не финальный redesign всей игры.

---

## 4. Базовые принципы новой боевой модели

### 4.1. Один боевой контракт для всех действий
Любое боевое действие должно укладываться в общую модель:
- у действия есть источник;
- у действия есть цель;
- у действия есть тип;
- у действия есть расчёт силы;
- у действия есть набор последствий.

### 4.2. Один turn pipeline
Один ход должен проходить через единый набор фаз.
Нельзя, чтобы normal attack и skill action жили как разные мини-игры.

### 4.3. Эффекты — это система, а не россыпь флагов
Баффы, дебаффы, DoT, триггеры и защитные эффекты должны иметь единый формат представления.

### 4.4. Сначала логика, потом цифры
Сначала фиксируем:
- когда что тикает;
- когда уменьшается duration;
- когда проверяется смерть;
- когда может сработать resurrection;
- как определяется enemy response.

И только после этого имеет смысл глубже трогать баланс.

### 4.5. Подготовка к будущему групповому бою
Даже если Combat Core v1 ещё не реализует полноценный party combat,
его модель не должна мешать в будущем:
- вешать эффекты на союзника;
- отличать self-target от ally-target;
- различать живую и мёртвую цель;
- добавлять revive-логики без полной переписи ядра.

## 5. Единый turn pipeline

Ниже — целевая модель одного полного хода.

### Phase 0 — Turn setup
На старте хода движок определяет:
- кто активная сторона;
- кто цель;
- живы ли участники;
- можно ли продолжать бой;
- не был ли бой уже закончен до этого.

### Phase 1 — Start-of-turn effects
На этой фазе применяются эффекты, которые должны срабатывать **в начале хода**:
- poison;
- burn;
- bleed;
- regeneration;
- resource ticks;
- другие эффекты с `tick_phase = start_turn`.

После применения эффектов делается проверка:
- не умерла ли цель от start-of-turn тиков;
- не должен ли бой закончиться уже здесь.

### Phase 2 — Action declaration
Активная сторона объявляет действие.
На уровне ядра это не callback, а логический тип действия:
- `normal_attack`
- `skill`
- `flee`
- `use_potion`
- в будущем: `defend`, `guard`, `revive_ally`, и т.д.

### Phase 3 — Action resolution
Движок считает само действие.
На этом этапе normal attack и damage-skill должны идти через единый принцип:

1. базовая сила действия;
2. стат-скейлинг;
3. weapon / school scaling;
4. crit / hit / dodge / immunity rules;
5. защита / mitigation;
6. итоговый урон / лечение;
7. primary result.

### Phase 4 — Post-action triggers
После основного действия применяются вторичные последствия:
- on-hit эффекты;
- parry;
- reflect;
- counter;
- consume-эффекты вроде mark / envenom;
- conditional bonuses;
- наложение новых timed-эффектов.

### Phase 5 — Enemy response
Если:
- цель жива;
- бой не закончен;
- враг ещё способен действовать;

то враг проходит через ту же логику объявления и разрешения действия,
но уже как ответная сторона.

### Phase 6 — End-of-turn effects
В конце хода применяются эффекты, которые должны тикать именно здесь:
- уменьшение duration у части баффов;
- истечение one-turn эффектов;
- cleanup флагов;
- end-turn resource logic.

### Phase 7 — Resolution checks
В конце полного хода выполняются финальные проверки:
- смерть игрока;
- resurrection / anti-death trigger;
- смерть врага;
- победа;
- завершение боя;
- переход в следующий ход.

## 6. Целевая модель timed effects

### 6.1. Почему нужна новая модель
Сейчас часть эффектов хранится списком, а часть — в виде отдельных полей в `battle_state`.
Это неудобно поддерживать и почти невозможно масштабировать.

### 6.2. Минимальный формат эффекта
Каждый timed-эффект в новой модели должен иметь понятную структуру.
Минимально:

```python
{
    "id": "resurrection",
    "type": "trigger_buff",
    "target_side": "player",
    "target_ref": "self",
    "value": 50,
    "duration": 5,
    "tick_phase": "end_turn",
    "trigger": "on_death",
    "stacks": False,
    "source_skill_id": "resurrection",
}
```

### 6.3. Обязательные поля эффекта
Минимальный набор полей:
- `id` — идентификатор эффекта;
- `type` — тип (`dot`, `hot`, `buff`, `debuff`, `trigger_buff`, `control`, и т.д.);
- `target_side` — на ком эффект (`player`, `enemy`, в будущем `ally`, `party`);
- `target_ref` — точная цель (`self`, `target`, `ally_slot`, и т.д.);
- `value` — основная сила эффекта;
- `duration` — сколько ходов он живёт;
- `tick_phase` — когда он должен тикать (`start_turn`, `end_turn`, `on_trigger`);
- `stacks` — стакуется ли;
- `source_skill_id` — откуда эффект пришёл.

### 6.4. Дополнительные поля по необходимости
Если нужно, эффект может иметь:
- `max_stacks`
- `trigger`
- `tags`
- `school`
- `can_dispel`
- `consumed_on_trigger`
- `apply_limit`
- `metadata`

---

## 7. Правила длительностей

Это один из самых важных разделов.

### 7.1. Duration — это реальные ходы, а не просто флаг
Если в описании скилла написано, что эффект длится N ходов,
то у него в runtime должен быть отдельный duration-счётчик.

### 7.2. У каждого эффекта есть явная точка тика
Эффект обязан знать, когда он живёт и когда уменьшается:
- `start_turn`
- `end_turn`
- `on_trigger`

### 7.3. Одноразовые эффекты живут как trigger-buff
Например:
- parry;
- guaranteed crit;
- resurrection;
- anti-death shield.

Они могут иметь duration, но срабатывают через trigger,
а не как обычный DoT/HoT.

### 7.4. Истечение эффекта должно быть предсказуемым
Нужно избегать ситуации, когда эффект живёт “примерно пока не забудем его снять”.

## 8. Resurrection: зафиксированное решение на этом этапе

### 8.1. Что НЕ делаем сейчас
Не делаем отдельный hotfix старой реализации resurrection ради косметики.

### 8.2. Что фиксируем в новой модели
`resurrection` в Combat Core v1 считается **timed trigger effect**.

### 8.3. Соло-логика
В соло он работает так:
- эффект вешается на себя;
- у эффекта есть duration;
- если носитель умирает до истечения duration — эффект срабатывает;
- цель встаёт с заданным % HP;
- после срабатывания эффект исчезает;
- если duration истёк и смерть не произошла — эффект просто пропадает.

### 8.4. Подготовка к группе
В будущей модели группового боя ядро должно поддерживать **оба варианта**:
1. бафф на живую цель, который поднимает её при смерти;
2. отдельное активное действие “поднять мёртвого союзника”.

### 8.5. Предварительное балансное предпочтение
Для первой групповой версии менее имбалансным считается вариант:
- **повесить timed anti-death buff на живую цель**.

Мгновенное поднятие уже мёртвого союзника обычно сильнее раскачивает бой
и требует более жёстких ограничений.

Итог: на уровне ядра нужно поддержать оба варианта,
но для первой версии эффекта безопаснее ориентироваться на anti-death buff.

## 9. Единая модель action resolution

### 9.1. Главная идея
Normal attack и damage-skill должны считаться через один общий боевой принцип.

### 9.2. Базовый порядок расчёта
Любое действие, наносящее урон, проходит такие этапы:

1. определить тип действия;
2. определить базовую силу;
3. применить скейлинг от статов;
4. применить modifiers от оружия / школы / ветки;
5. применить offensive bonuses;
6. проверить crit / hit / dodge / immunity;
7. применить defense / mitigation / penetration;
8. получить final damage;
9. применить пост-эффекты.

### 9.3. Почему это важно
Это позволит:
- не дублировать полбойцовской логики в двух местах;
- сделать формулы понятнее;
- легче менять баланс;
- легче тестировать боевой расчёт.

---

## 10. Границы ответственности между файлами

### `handlers/battle.py`
Должен отвечать только за:
- Telegram callback/UI;
- orchestration;
- сохранение/очистку состояния;
- вызов движка;
- показ результата.

### `game/combat.py`
Должен отвечать за:
- turn pipeline;
- phase resolution;
- enemy response;
- ticking эффектов;
- death/resurrection/victory checks;
- общий боевой расчёт.

### `game/skill_engine.py`
Должен отвечать за:
- описание того, что делает skill;
- формирование боевого действия или эффекта;
- mana cost / cooldown / targeting rules;
- передачу в combat engine параметров действия.

### `game/balance.py`
Должен отвечать за:
- формулы статов;
- формулы offensive/defensive scaling;
- caps / soft caps;
- общую математическую базу.

## 11. Порядок дальнейшей реализации

### Этап A — спецификация
Сначала фиксируем этот документ как рабочую модель.

### Этап B — core architecture
Потом в коде делаем:
1. новую timed-effect модель;
2. единый action contract;
3. единый turn pipeline;
4. перенос логики из handler-слоёв в combat core.

### Этап C — formulas
Потом уже отдельно:
- переписка / нормализация `balance.py`;
- выравнивание базовых формул;
- настройка caps.

### Этап D — weapons and skills redesign
И только после этого:
- новые оружия;
- новые ветки;
- новая философия архетипов;
- доработка skill trees под `GAME_FOUNDATION.md`.

---

## 12. Что считать успехом Combat Core v1

Combat Core v1 считается успешно зафиксированным и реализуемым, если:

1. есть понятный единый pipeline одного хода;
2. timed-эффекты больше не живут как хаотичный набор флагов;
3. resurrection и похожие эффекты имеют нормальную duration-модель;
4. normal attack и damage-skill используют единый боевой контракт;
5. `handlers/battle.py` не содержит критической боевой математики;
6. дальнейшая перепись `balance.py` и оружий может идти уже поверх нормального ядра.

## 13. Рекомендация по структуре документации

Есть хорошая идея собрать информационные документы в одну папку,
но тут есть важная оговорка:

### `AGENTS.md` лучше оставить в корне репозитория
Причина:
- этот файл является главным инструкционным файлом для coding agents;
- его расположение в корне — ожидаемая и удобная точка входа.

### Остальные информационные документы логично унести в `docs/`
Рекомендуемый вариант:

```text
rpg/
├── AGENTS.md
├── docs/
│   ├── CLAUDE.md
│   ├── GAME_FOUNDATION.md
│   ├── COMBAT_CORE_V1_SPEC.md
│   └── ... другие проектные спецификации
```

### Допустимое развитие структуры позже
Если документов станет много:

```text
rpg/
├── AGENTS.md
├── docs/
│   ├── foundation/
│   │   └── GAME_FOUNDATION.md
│   ├── architecture/
│   │   ├── CLAUDE.md
│   │   └── COMBAT_CORE_V1_SPEC.md
│   ├── balance/
│   └── roadmap/
```

Но прямо сейчас, чтобы не усложнять, достаточно простого `docs/`.

## 14. Практическое правило для будущих изменений

Если в будущем ломается боёвка или начинается новая фаза переписки,
нужно сверяться с этим документом в таком порядке:

1. Это изменение относится к pipeline, эффектам или формулам?
2. Не ломает ли оно единый контракт normal attack / skill action?
3. Не создаёт ли оно новый special-case мимо общей модели?
4. Не вводит ли оно эффект без нормальной duration-модели?
5. Не тащит ли оно боевую логику обратно в handler-слой?

Если ответ “ломает” — значит, изменение надо пересмотреть.

## 15. Итог

Combat Core v1 — это не финальный баланс и не полный redesign игры.

Это **фундамент**, который нужен, чтобы:
- проект не рассыпался от будущих изменений;
- оружия и навыки можно было спокойно переделывать;
- timed-эффекты жили предсказуемо;
- боевую систему можно было расширять без постоянной борьбы с наследием старого кода.
